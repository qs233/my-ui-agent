# 可见性过滤计算

本文说明当前算法中和可见性过滤相关的 CSS 属性如何分类，以及为什么有的属性只看当前节点，有的属性需要递归检查祖先节点。

相关实现：

- `src/prepare.ts` 的 `isRetainedElement`
- `src/prepare.ts` 的 `isElementRenderBlocked`
- `src/prepare.ts` 的 `isTextRenderBlocked`

## 总体判断标准

核心标准不是“父节点有没有设置某个属性”，而是：

**祖先对后代的影响，能不能从后代自己的 computed style 中可靠看出来。**

如果能从当前节点的 computed style 或 layout 结果中判断，就只看当前节点。  
如果祖先会影响整棵子树的最终渲染，但这个影响不一定体现在子节点自己的 computed style 上，就需要沿父链递归检查。

## 只看当前节点的属性

### `display`

当前算法只直接过滤当前元素的：

```css
display: none;
```

判断位置：

```ts
if (element.styles.get("display") === "none") return false;
```

原因：

- `display: none` 会让元素不产生 layout box。
- 如果祖先是 `display: none`，子树通常不会出现在 Chrome DOMSnapshot 的 layout 表里，也就不会进入后续 retained 判断。
- 如果当前元素自己的 computed `display` 是 `none`，直接过滤即可。

因此 `display` 不需要像 `opacity` 那样递归检查祖先。

### `visibility`

当前算法过滤：

```css
visibility: hidden;
visibility: collapse;
```

判断位置：

```ts
const visibility = element.styles.get("visibility");
if (visibility === "hidden" || visibility === "collapse") return false;
```

原因：

- `visibility` 是继承属性。
- 父元素设置 `visibility: hidden` 时，子元素的 computed `visibility` 通常也会变成 `hidden`。
- 子元素可以显式覆盖为 `visibility: visible`，并重新显示出来。

例如：

```html
<div style="visibility: hidden">
  <button>Submit</button>
</div>
```

这里 `button` 的 computed `visibility` 通常就是 `hidden`，看当前节点即可过滤。

但这个例子不能被父链递归误杀：

```html
<div style="visibility: hidden">
  <button style="visibility: visible">Submit</button>
</div>
```

这里 `button` 可以显示。如果递归检查祖先并把父级 `visibility: hidden` 当成整棵子树不可见，就会错误过滤这个按钮。

所以 `visibility` 的判断标准是：**看当前节点 computed visibility 的最终结果，而不是简单递归父级。**

## 需要递归检查祖先的属性

### `opacity`

当前算法递归过滤：

```css
opacity: 0;
```

判断位置：

```ts
const opacity = Number(element.styles.get("opacity") ?? "1");
const blocksSelf = opacity === 0;
```

原因：

- `opacity` 不继承。
- 子元素 computed `opacity` 可能仍然是 `1`。
- 但浏览器绘制时会把父元素及其子树作为整体应用透明度。

例如：

```html
<div style="opacity: 0">
  <button style="opacity: 1">Submit</button>
</div>
```

`button` 自己的 computed `opacity` 是 `1`，但因为祖先整体透明，按钮实际不可见。  
所以必须递归检查祖先：

```ts
const blocked =
  blocksSelf ||
  (parent ? isElementRenderBlocked(parent, elements, memo) : false);
```

### `content-visibility`

当前算法递归过滤：

```css
content-visibility: hidden;
```

判断位置：

```ts
const blocksSelf =
  opacity === 0 ||
  element.styles.get("content-visibility") === "hidden";
```

原因：

- `content-visibility: hidden` 会让该元素的内容跳过渲染。
- 这个影响作用在子树层面。
- 子节点自己的 computed style 不一定能直接体现祖先的这个渲染阻断。

因此它和 `opacity: 0` 一样，需要沿父链检查。

## 元素节点的过滤路径

元素节点通过 `isRetainedElement` 判断是否保留：

1. 过滤宽高为 0 的元素。
2. 过滤当前元素 `display: none`。
3. 过滤当前元素 `visibility: hidden/collapse`。
4. 递归检查自己或祖先是否存在 `opacity: 0` 或 `content-visibility: hidden`。

简化后是：

```ts
if (element.bounds.width <= 0 || element.bounds.height <= 0) return false;
if (element.styles.get("display") === "none") return false;
if (visibility === "hidden" || visibility === "collapse") return false;
return !isElementRenderBlocked(element, elements, renderBlockedMemo);
```

## 文本节点的过滤路径

文本节点没有自己的 computed style，因此不能像元素节点一样直接检查 `display`、`visibility`、`opacity`。

当前算法先判断文本自身是否有内容和面积：

```ts
normalizeText(node.text).length > 0 &&
node.bounds.width > 0 &&
node.bounds.height > 0
```

然后通过最近的父元素判断是否被阻断：

```ts
const nearestElement = text.parentElementNodeIndex === null
  ? undefined
  : elements.get(text.parentElementNodeIndex);
```

判断逻辑：

1. 找不到最近父元素，则认为文本不可用。
2. 最近父元素 `visibility: hidden/collapse`，则过滤。
3. 最近父元素或其祖先存在 `opacity: 0` / `content-visibility: hidden`，则过滤。

## 分类总结

| 分类 | 属性 | 判断方式 | 原因 |
| --- | --- | --- | --- |
| 当前节点即可判断 | `display` | 看当前元素 computed `display` 和 layout 是否存在 | 祖先 `display: none` 的子树通常不会进入 layout 表 |
| 当前节点即可判断 | `visibility` | 看当前元素 computed `visibility` | 它是继承属性，且子元素可以覆盖回 `visible` |
| 需要递归祖先 | `opacity` | 检查自己和祖先是否有 `opacity: 0` | 不继承，但祖先透明会让整棵子树不可见 |
| 需要递归祖先 | `content-visibility` | 检查自己和祖先是否有 `content-visibility: hidden` | 祖先会让子树跳过渲染，子节点 computed style 不一定体现 |
